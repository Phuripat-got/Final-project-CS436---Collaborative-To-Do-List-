# Design_Rationale.md: เอกสารอธิบายเหตุผลในการเลือกใช้ UX/UI
**Project:** Real-time Multi-user To-Do List Application

---

## 1. ปรัชญาการออกแบบ (Design Philosophy)
การออกแบบยึดหลัก **"Minimalism & Efficiency"** เนื่องจากเป็นเครื่องมือที่เน้นการใช้งานจริง (Utility-first) และต้องการความเร็วในการตอบสนอง (Real-time)
* **Simplicity:** ตัดสิ่งรบกวน (Distractions) ออกทั้งหมด เหลือไว้เฉพาะสิ่งที่จำเป็นต่องาน (Content-focused)
* **Clarity:** สถานะของงาน (เสร็จ/ไม่เสร็จ) และความเป็นเจ้าของงาน (ใครสร้าง) ต้องชัดเจนในเสี้ยววินาที
* **Invisible Interface:** ทำให้อินเทอร์เฟซดูธรรมดาที่สุด เพื่อให้ผู้ใช้โฟกัสไปที่ "ตัวงาน" ที่ต้องทำจริงๆ

---

## 2. การตัดสินใจด้านโครงสร้างและการจัดวาง (Layout Decisions)

### 2.1 Desktop-First Layout (1440px)
* **Rationale:** เลือกใช้ Layout มาตรฐาน Desktop เพราะ Use Case ของการทำงานร่วมกัน (Collaboration) มักเกิดขึ้นในสภาพแวดล้อมการทำงานบนหน้าจอคอมพิวเตอร์
* **Centered Container:** จำกัดความกว้างของเนื้อหาหลัก (Main Content) ไว้ที่ 800px ตรงกลางหน้าจอ เพื่อให้อ่านง่าย (Readability) สายตาไม่ต้องกวาดมองกว้างเกินไปบนจอ Wide-screen

### 2.2 Single-Page Application (SPA)
* **Rationale:** แยกหน้า Login (Identify) ออกจาก Dashboard อย่างชัดเจน แต่เมื่อเข้าสู่ Dashboard แล้ว ทุกฟังก์ชัน (Add, Edit, Delete, Sync) จะจบในหน้าเดียว เพื่อลด Interaction Cost ไม่ต้องกดเปลี่ยนหน้าไปมา

### 2.3 กลไกการจัดวางตามจิตวิทยา (Layout Mechanics)
* **F-Pattern Scanning:** จัดวางองค์ประกอบหน้า Dashboard เรียงจากบนลงล่าง (Header -> Input -> List) สอดคล้องกับธรรมชาติการกวาดสายตาของมนุษย์
* **Z-Layout (หน้า Login):** หน้า Login วางองค์ประกอบเรียบง่าย เพื่อนำสายตาผู้ใช้เป็นรูปตัว Z พุ่งเป้าไปที่ปุ่ม "Join Room" ทันที
* **White Space:** การเว้นระยะห่าง (Padding/Margin) อย่างเหมาะสม ช่วยลดภาระทางสมอง (Cognitive Load) ทำให้ผู้ใช้ไม่รู้สึกอึดอัดเมื่อมีงานจำนวนมาก

## 3. การตัดสินใจด้านสีและส่วนติดต่อผู้ใช้ (Color & UI Decisions)

### 3.1 ทฤษฎีสี (Color Theory) - กฎ 60-30-10
ใช้สัดส่วนสีเพื่อความสมดุลและความสวยงาม:
* **60% สีพื้น (White/Light Gray):** พื้นหลังและพื้นที่ว่าง เพื่อความสะอาดตาและขับเน้นเนื้อหา
* **30% สีรอง (Black/Dark Gray):** ตัวหนังสือและเนื้อหางาน (Task Title)
* **10% สีเน้น (Blue/Green/Red):** ใช้เฉพาะจุดที่โต้ตอบได้ (Interactive Areas) เช่น ปุ่ม, Checkbox

### 3.2 จิตวิทยาสี (Color Psychology)
* **Blue (#3B82F6):** สื่อถึงความสงบ สมาธิ และความน่าเชื่อถือ (Productivity) ใช้เป็นสีหลักของแบรนด์และปุ่ม Action
* **Green (#10B981):** สื่อถึงความสำเร็จ ใช้กับ Checkbox เมื่อทำงานเสร็จ เพื่อให้รางวัลทางความรู้สึก (Positive Reinforcement)
* **Red (#EF4444):** สื่อถึงอันตราย ใช้กับปุ่ม Delete เพื่อเตือนสติก่อนลบ

### 3.3 การเข้าถึง (Accessibility)
* **Contrast Ratio:** สีตัวอักษรกับพื้นหลังมีค่าความต่างแสงอย่างน้อย 4.5:1 (มาตรฐาน WCAG AA) เพื่อให้ผู้ที่มีปัญหาทางสายตาอ่านได้ชัดเจน
* **Typography:** เลือกใช้ฟอนต์ **Sans-serif** (เช่น Inter, Roboto) ที่ไม่มีหัว อ่านง่ายบนหน้าจอดิจิทัล ดูทันสมัย

---

## 4. การตัดสินใจด้านประสบการณ์ผู้ใช้และกระแสการใช้งาน (UX & Flow)

### 4.1 Real-time Feedback Loop
* **Design:** **ไม่มีปุ่ม "Refresh" หรือ "Sync"** บนหน้าจอ
* **Rationale:** เพื่อสร้างความเชื่อมั่น (Trust) ว่าระบบทำงานแบบ Real-time อัตโนมัติ ลดภาระที่ผู้ใช้ต้องคอยกดโหลดข้อมูลเอง

### 4.2 สถานะของงาน (Status Indication)
* **Design:** เมื่อ Task เสร็จสิ้น (Completed) จะเกิด 3 สิ่งพร้อมกัน:
    1.  Checkbox เป็นสีเขียว
    2.  ข้อความจางลง (Opacity 50%)
    3.  มีเส้นขีดฆ่า (Strikethrough)
* **Rationale:** เพื่อให้ผู้ใช้แยกแยะงานที่ต้องทำ vs งานที่เสร็จแล้ว ได้ทันทีจากการกวาดสายตา (Scan)

### 4.3 บริบทความเป็นเจ้าของ (Ownership Context)
* **Design:** แสดงข้อความ "Created by: [Name]" ขนาดเล็กใต้ชื่องาน
* **Rationale:** ในระบบ Multi-user การระบุตัวตนผู้สร้างงานช่วยสร้างความเกรงใจและลดความขัดแย้ง (Conflict) ในการลบหรือแก้ไขงานของผู้อื่น

### 4.4 Happy Path Flow
* **Design:** เน้นการใช้งานด้วยคีย์บอร์ด (Keyboard First) เช่น พิมพ์งาน -> กด Enter เพื่อบันทึกทันที
* **Rationale:** เพื่อความรวดเร็วและความต่อเนื่องในการใช้งาน (Flow State) โดยไม่ต้องสลับไปใช้เมาส์บ่อยๆ

---

## 5. การจัดการข้อจำกัด (Handling Constraints)
* **Conflict Resolution:** กรณีมีการแก้ไขงานเดียวกันพร้อมกัน UI จะยึดตามข้อมูลล่าสุด (Last Write Wins) โดยไม่มี Pop-up แจ้งเตือนที่ซับซ้อน เพื่อรักษาความลื่นไหลของระบบ (Simplicity over Complexity) ตามขอบเขตของ MVP
